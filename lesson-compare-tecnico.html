Flujo de ejecución

1

initSelects() — mapea PARTICIPANTS a <option> elements en los 3 <select>. Se llama una vez al cargar.

2

onchange → render() — lee los 3 selects, busca cada participante en PARTICIPANTS, filtra MATCHES según showPending.

3

classifyBet(bet, result) — devuelve "exact" | "winner" | "wrong" | "none". Se usa como sufijo de clase CSS: bet-exact, bet-winner, etc.

4

La tabla se construye como un template string de HTML y se vuelca en wrap.innerHTML — re-render completo en cada cambio de select o toggle.

El corazón: classifyBet + calcPts

bets-compare.html — lógica central
function classifyBet(bet, result) { if (!result) return "none"; // partido pendiente const [bh, ba] = bet; const [rh, ra] = result; if (bh === rh && ba === ra) return "exact"; // marcador exacto if (Math.sign(bh-ba) === Math.sign(rh-ra)) return "winner"; // ganador correcto return "wrong"; } function calcPts(bet, result, phase) { if (!result) return null; // null ≠ 0 — pendiente vs incorrecto const mult = MULT[phase] || 0.33; const [bh, ba] = bet; const [rh, ra] = result; if (bh === rh && ba === ra) return +(3 * mult).toFixed(2); if (Math.sign(bh-ba) === Math.sign(rh-ra)) return +(1 * mult).toFixed(2); return 0; }

En calcPts(), ¿por qué se retorna null cuando !result, en vez de 0?

  • A) Porque 0 causaría un error al sumar el total de puntos
  • B) Para distinguir "partido no jugado" de "apuesta incorrecta" — ambos darían 0 sin esta diferencia
  • C) Porque Reveal.js no puede renderizar el número 0 en celdas de tabla

Presioná → para ver la respuesta

  • ✓ Correcto: B — null permite que la UI muestre "⏳" (pendiente) vs "0pt" (jugado y errado). Sin esa distinción, las celdas pendientes parecerían incorrectas.
💡 Recordá: null = "aún no hay dato" — semánticamente distinto de 0 = "el dato existe y es cero".
