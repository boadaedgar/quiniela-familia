What Is It

How @playwright/mcp bridges AI agents and real browsers through structured accessibility data — and why that beats screenshots.

↓ to dive in
01 — Core Concept

The Core Concept

@playwright/mcp is the official Microsoft MCP server that gives any AI agent structured control over a real web browser. It wraps the Playwright engine in Model Context Protocol tools — so AI clients can navigate, click, fill forms, and inspect pages without writing a line of code.

Key insight: The AI doesn't see pixels — it reads a structured accessibility snapshot. Each interactive element gets a unique ref. The model picks the right ref and acts on it deterministically.

Supported browsers: Chrome · Firefox · WebKit · Edge — one config, any engine.

01 — Accessibility Tree vs Screenshots

Accessibility Tree vs. Screenshots

❌ Screenshot approach

  • Vision model reads raw pixels
  • 2,000 – 10,000+ tokens per frame
  • Coordinate guessing — brittle on resize
  • Requires vision-capable model
  • Breaks on dynamic / animated layouts

✅ Accessibility tree

  • Structured text: role, name, ref
  • 200 – 400 tokens per snapshot
  • Deterministic refs — reliable
  • Works with any language model
  • Layout-independent
Result: 10–50× cheaper per action. No vision model needed. Works with Claude, GPT-4o, Gemini, or any MCP-compatible agent.
01 — Architecture

How It Fits Together

1
User gives AI client a natural-language task
"Navigate to github.com and find the playwright/playwright repo star count"
2
AI client calls @playwright/mcp tools via MCP protocol
browser_navigate → browser_snapshot → browser_click → …
3
MCP server drives a local browser with Playwright
Chrome / Firefox / WebKit / Edge — your choice
4
Accessibility snapshot returned to AI — loop continues
Model reads refs, decides next action, calls next tool

02 — Prerequisites

Prerequisites

  • Node.js 18+ — the only hard requirement; check with node --version
  • An MCP client — Claude Code, VS Code Copilot, Cursor, Windsurf, Claude Desktop, Cline, Goose, Kiro, Codex, and others
  • No global install needed — npx @playwright/mcp@latest fetches it on first run
  • Browsers download automatically via Playwright on first use
Warning: The correct package is @playwright/mcp (scoped, by Microsoft). The unscoped playwright-mcp on npm is a different community package — easy to install by mistake.

Install by MCP Client

02 — Universal JSON Config

Universal JSON Config

Use this in any MCP client that accepts a settings JSON (Cursor, Windsurf, Claude Desktop, etc.):

{ "mcpServers": { "playwright": { "command": "npx", "args": ["@playwright/mcp@latest"] } } }
Verify: Restart your MCP client → look for Playwright tools in the tool list → prompt: "Take a screenshot of example.com" to confirm.

Section 4 — Workflows

04

Workflows

Four common patterns: basic web task, login + session persistence, form automation, and AI web research.

↓ to dive in
04 — Basic Web Task

Basic Web Task

1
Navigate to the target URL
browser_navigate → "https://example.com"
2
Take an accessibility snapshot
browser_snapshot → returns tree with element refs (buttons, links, inputs)
3
Act on the target element using its ref
browser_click ref="e15" → clicks the "Sign in" button
4
Snapshot again to confirm state change
browser_snapshot → verify page transitioned as expected
Core pattern: navigate → snapshot → act → snapshot (repeat until done)
04 — Login Flow

Login Flow + Session Persistence

1
Navigate to login page
browser_navigate → "https://app.example.com/login"
2
Snapshot to get input refs
browser_snapshot → identifies email field ref="e3", password ref="e4"
3
Fill credentials and submit
browser_fill ref="e3" → "user@example.com"
browser_fill ref="e4" → "password"
browser_click ref="submit-btn"
4
Session persists automatically between agent turns
Cookies + localStorage saved — no re-auth needed for rest of session
04 — AI Web Research

AI Agent Web Research

1
User gives agent a research task
"Search for '@playwright/mcp' and summarize the top 3 results"
2
Agent navigates and fills search
browser_navigate → google.com → browser_fill search box → browser_key_press "Enter"
3
Snapshot extracts result titles and links
browser_snapshot → agent reads result titles, URLs, descriptions from accessibility tree
4
Agent visits each page and synthesizes
browser_navigate (×3) → browser_snapshot → agent writes summary
Tip: Snapshots keep research loops cheap. Use browser_screenshot only when you need visual proof.
04 — Network Mocking for Testing

Network Mocking for Testing

1
Set up mock before navigating
browser_mock_api url="/api/orders" → return fake order data
2
Navigate to the page that calls the API
browser_navigate → "https://app.example.com/orders"
3
Snapshot to verify page rendered mocked data
browser_snapshot → confirm mock order list appears in accessibility tree
4
Inspect network requests to audit traffic
browser_network_requests → verify only expected endpoints were called

05

Limits & Tips

Known constraints, common gotchas, and pro tips for reliable AI browser automation.

↓ to dive in
05 — Known Limits

Known Limits

LimitWhy it happensWorkaround
Sparse accessibility trees Sites suppress ARIA roles or use canvas-heavy UIs Fall back to browser_screenshot + vision model
Web only No native desktop UI automation support Use Anthropic computer-use for desktop targets
Node.js 18+ required Hard dependency on local Node runtime Use Browserbase cloud if you can't install Node
Single session per instance One browser window per MCP server process Run multiple MCP server instances for parallel tasks
Package name confusion Unscoped playwright-mcp on npm is a different package Always use @playwright/mcp (scoped)
05 — Pro Tips

Pro Tips

  • Always snapshot before acting — element refs change after navigation; stale refs fail silently
  • Use browser_fill, not browser_type — fill replaces the existing value cleanly; type appends
  • Set up browser_mock_api before navigating — intercepts must be registered before the page loads
  • Start a trace before long workflows — browser_start_trace gives you a full debug replay on failure
  • Prefer snapshots over screenshots — snapshots are 10–50× cheaper; use screenshots only for visual proof
  • Session persists by default — log in once; subsequent agent turns reuse cookies automatically
  • Pin a version for production — use @playwright/mcp@1.x.x instead of @latest in CI
